## Bugs found + fixes shipped

### Fixed (commit e2e2f40c)

**`lib/std/hash.aura` — self-recursive `hash-ref` shadow.**

```
(define (hash-ref h k)
  (hash-ref h k))     ; ← body refers to the local binding, not the primitive
```

When `std/hash` was required, this shadowed the C++ primitive with a
self-recursive function. Every importer hit
`error: ...: unbound variable: k` on the first `hash-ref` call.

Fix: delete the redundant redefinition. `hash-ref` resolves to the C++
primitive in the importer env. `hash-get` (which calls `hash-ref`) now
works through the primitive path.

Verified: `./build/aura < /tmp/aura_hash_fix_verify.aura` returns
`ref a=1 / ref b=2 / len=2` (vs error before).

### Found, NOT fixed — follow-up issues to file

These are deeper `evaluator_eval_flat.cpp` bugs. Surface test pins them
down; the fix is non-trivial C++ work and would need its own session.

1. **`(set! var val)` fails inside function bodies.**
   Works at top-level. Inside `(define (f x) (set! ...))`, raises
   `unbound variable: set!:`. Repro: `tests/commercial_readiness_stdlib_surface_2026_07_17.aura`
   exhibits this in every stdlib helper that uses local mutation. Affects
   ~half the stdlib.

2. **Anonymous lambda passed as a function arg fails with
   `invalid closure: eval_flat: invalid closure`.**
   Named functions work (probe `(apply-it inc 41) = 42`); anonymous
   lambdas don't (probe `(apply-it (lambda (y) (+ y 1)) 41)` errors).
   Blocks `map`/`foldr`/`filter` from stdlib at the user-facing layer.

3. **Named recursive function: `if`'s else-branch runs even when cond is
   true.**
   `(define (count n) (if (= n 0) (begin (display "BASE") 999) (begin
   (display "REC") (count (- n 1)))))` with `(count 0)` prints "REC"
   (else-branch side effect) and returns `()` — should print "BASE" and
   return 999. Likely a deeper eval_flat routing bug, possibly related
   to #2.

4. **`alist->hash` with `(require "std/hash" all:)` doesn't expose.**
   The `all:` form fails with `unbound variable: alist`. Without
   `all:` it works. Probably `all:` parsing/multi-line export format
   issue in the require primitive.

## Process notes
