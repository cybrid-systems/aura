## #1397 - cells_/pairs/string_heap_ uniform alloc_storage_lock (Option A)

### Root cause

`fiber:spawn` shares the Evaluator between spawning thread and worker thread. Without serializing push_back on `cells_` / `pairs_` / `string_heap_`, the race produces lost updates, iterator invalidation mid-iter, torn pair slots (racy `size()-1` index returned by `push_back` can be reallocated before caller uses it).

- Added `mutable std::mutex alloc_storage_lock_` to `Evaluator` class (public, so `register_*_primitives` free functions in `evaluator_primitives_*.cpp` can acquire it).

## #1398 - hash-set! resize + hash-keys/hash-values iter invalidation contract test (Anqi said "continue #1398" at 17:41)

### Scope (per #1398 body)

`hash-set!` triggers internal bucket array reallocation on load-factor threshold. If `hash-keys` / `hash-values` returned a live view (rather than a copy), the iterator's bucket pointer would be invalidated and user-visible: dangling pair list.

The existing implementation already creates freshly-consed pair lists via `EvalValue{keys[i-1]}` / `EvalValue{vals[i-1]}` copies - so `hash-keys` / `hash-values` / `hash->alist` return BY VALUE (per-call snapshot, not a live view into the hash's internal storage).

### Operational notes (added this session)

- The framework's 3-asterisk substitution in `python3 << 'PYEOF' ... PYEOF` heredoc is unreliable - it sometimes substitutes the 3-asterisk pattern to something that produces invalid Python syntax (e.g., unmatched `]`). Workaround: `cat ~/.github-token | python3 -c "import os; T = os.environ['TOKEN']; ..."` reads token via env var, avoiding the substitution entirely. Or use `with open('/home/dev/.github-token') as f: T = f.read().strip()` in a file-based script (not heredoc) to sidestep the substitution entirely.
