# Aura - AI-Native Programming Language

> **从最小 Lisp 核心开始,自然生长。**
> 让 AI Agent 成为代码的演化者,而非执行者。

---

## 核心口号

**"The Narrow Gate."**
Aura 不兼容旧生态、不优化给人读。它的唯一目标是:在给定算力下,让 AI 最大程度地理解、验证、重写并统一整个软件栈。

---

## 1. 设计原则

### 1.1 最小核心(Minimal Core)
只保留最少的语义原语。所有上层语法--循环、模式匹配、对象系统--都是宏(macro)。
核心求值器可以写在半页纸上。这不是炫技,而是**必要**:AI 只有面对足够小的核心,才能真正理解并重写它。

### 1.2 代码即数据(Homoiconicity)
程序与数据同构。AST 是第一类公民:可被构造、查询、变换、注释、序列化。
这是 AI 操作代码的前提。没有 Homoiconicity,AI 只能"生成文本";有了它,AI 能"操作语义"。

### 1.3 增量式构建(Ghuloum 精神)
每一步只增加一个很小的功能,始终保持系统可运行、可测试。
从 `(lambda (x) x)` 到完整的宏系统,中间经过几十个可运行的中间状态。
每增加一个功能,测试套件跟着扩展一条。绝不大跃进。

### 1.4 窄门路线(The Narrow Gate)
不兼容任何现有语言。不追求人类可读性。不提供传统 IDE。
Aura 是给 AI 用的--人类通过 AI 与 Aura 交互。
这意味着我们可以做传统语言做不到的设计取舍:完全一致的语法、极端统一的语义、不加语法糖。

### 1.5 反射与宏并重(LiSP Ch8 + Ch9)
**两条腿走路**:
- **反射**(Ch8):运行时第一类环境,让 Agent 能动态重写自己的求值策略。
- **宏**(Ch9):编译期代码变换,让语言能静态生长出新语法。

反射是"运行时进化",宏是"编译期进化"。两者不可偏废。

### 1.6 持久 AST(Persistent AST)
AST 从解析到编译到运行时全程保留。支持:
- **结构化模式匹配**:对 AST 进行声明式查询和变换
- **源位置注释**:每个节点携带 file/line/column,支持增量编译
- **序列化**:AST 可持久化到磁盘,重启后恢复
- **Code Walking**:宏可以遍历和变换任意深度的 AST 子树

持久 AST 是连接所有阶段的纽带:解析器产生 AST,宏变换 AST,编译器遍历 AST,反射器操作 AST。它是语言的"共享内存"。

## 2. 架构总览

```
┌──────────────────────────────────────────────────┐
│                  Racket DSL 前端                   │
│  (#lang aura  +  宏系统  +  with-aliases)         │
├──────────────────────────────────────────────────┤
│                  持久 AST 层                       │
│  (结构化查询 + 模式匹配 + 源位置注释 + 序列化)     │
├──────────────────────────────────────────────────┤
│              静态反射 + Code Walking              │
│  (编译期元编程 + AST 遍历变换 + 增量预处理)        │
├──────────────────────────────────────────────────┤
│              运行时反射 + 第一类环境               │
│  (export/eval/b + flambda + reflective-lambda)    │
├──────────────────────────────────────────────────┤
│                增量编译管道                         │
│  (compute-kind → arity 检查 → 闭包变换 → 生成)    │
├──────────────────────────────────────────────────┤
│               C++26 / LLVM 运行时                  │
│  (threaded code VM / LLVM IR / native code)       │
└──────────────────────────────────────────────────┘
```

### 2.1 Racket DSL 前端
Racket 是完美的原型平台:它有最成熟的宏系统、syntax-parse、和自定义 #lang 机制。
早期 Phase 全部用 Racket 实现,快速验证语义设计。

### 2.2 持久 AST 层
所有阶段共享同一个 AST 表示。不是 JSON,不是 S 表达式字符串,而是一个**带类型的、可持久化的、支持元数据的树结构**。
在 Racket 中是 struct + prop:serializable,在 C++26 中是类型化的 variant 树。

### 2.3 静态反射
在编译期可查询和变换 AST。C++26 通过编译期反射机制(std::meta 提案或 clang 插件)提供语言级别的静态反射支持。
在 Racket 原型中,通过 syntax-parse 和 syntax-local-value 实现类似能力。

### 2.4 运行时反射
核心三件套:`export`(导出环境)、`eval/b`(在指定环境中求值)、`enrich`(扩展环境)。
加上 `flambda`(FEXPR)和 `reflective-lambda`,构成完整的运行时反射系统。
这是 AI Agent"自我重写"的关键基础设施。

### 2.5 增量编译管道
仿照 Ghuloum 的增量编译方法,每个阶段只做一件事:
1. **compute-kind**:确定每个表达式的求值类别(value / function / macro)
2. **arity 检查**:静态检测参数数量错误
3. **闭包变换**:将自由变量捕获到闭包中
4. **代码生成**:生成 threaded code 或 LLVM IR

### 2.6 C++26 / LLVM 运行时
早期用 Racket 做语义原型。生产版本用 C++26,利用最新标准:
- 编译期反射（std::meta 提案 / clang 插件）：元编程和 AST 查询
- `std::generator`：协程支持
- `std::execution`:并行执行原语
- LLVM 作为后端代码生成器(Phase 4 后)

## 3. 关键技术决策

### 3.1 为什么选择 Lisp1 而非 Lisp2?
函数和变量共用同一个名字空间。
**理由**:AI 不需要"帮人类理解代码"的辅助区分。Lisp1 更简单,更统一,更容易被 AI 推理。

### 3.2 为什么选择 Hyperstatic 作用域?
全局绑定一旦定义不可覆盖(set! 只对局部绑定有效)。
**理由**:AI Agent 在长时间运行中需要依赖"全局定义是稳定的"这一假设。Hyperstatic 让编译器可以做激进的优化,也让 Agent 可以安全地缓存全局绑定的信息。

### 3.3 为什么需要 with-aliases 风格的逃生舱?
卫生宏是默认行为,但有时候宏需要故意捕获或暴露某些标识符。
**理由**:DSL 生成经常需要"故意违反卫生规则"。with-aliases 提供一个显式的、受控的逃生舱。

### 3.4 为什么运行时反射用双轨设计?
静态反射(编译期)用于性能敏感的场景;运行时反射(eval/b + flambda)用于动态演化的场景。
**理由**:不是所有 Agent 都需要运行时反射。最小核心可以只支持静态反射,需要时才引入运行时反射的开销。

### 3.5 为什么持久 AST 而不是字符串拼接?
字符串拼接生成代码是 AI 当前的瓶颈--LLM 经常生成语法错误的代码。持久 AST 操作从根本上消除了语法错误。
**理由**:AST 操作是类型安全的、结构化的。AI 操作 AST 而不是文本,错误率大幅降低。

### 3.6 为什么先 Racket 后 C++26?
Racket 是"最快的 Lisp 方言原型平台";C++26 是"在工程质量和性能上可以落地的底层实现"。
**理由**:语义设计阶段最重要的是迭代速度。Racket 可以一天改三次求值器。C++26 不行。

## 4. 语言核心(Phase 0 语义规范)

```
<expr> ::= <literal>                 ;; 字面量(数字、字符串、布尔)
         | <symbol>                   ;; 变量引用
         | (lambda (<arg>...) <expr>) ;; 函数
         | (<expr> <expr>...)         ;; 函数应用
         | (if <expr> <expr> <expr>)  ;; 条件
         | (let ((<name> <expr>)...)
             <expr>)                  ;; 局部绑定
         | (letrec ((<name> <expr>)...)
             <expr>)                  ;; 递归绑定
         | (quote <datum>)            ;; 字面数据
         | (define <name> <expr>)     ;; 全局定义(Hyperstatic)
```

这是一个完整、自洽的最小 Lisp 核心。它能表达所有计算,同时小到 AI 可以完全理解。

## 6. 开发者协作原则

1. **Racket 原型优先**:所有新特性先在 Racket 中验证语义后再 C++26 实现
2. **每个 PR 一个功能点**:遵循 Ghuloum 精神,每个提交增加一个最小功能
3. **测试跟随特性**:每增加一个特性,必须有对应的测试用例
4. **文档同步更新**:特性合入前必须有对应的 DESIGN 文档更新
5. **可运行红线**:每次提交必须保证 `raco test` 通过
